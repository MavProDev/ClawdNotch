## Audit Summary

Full deep-dive audit of every line in the codebase (5,262 lines production, 1,426 lines tests, all CI/CD, configs, docs). Identified 16 issues ranging from critical bugs to low-priority polish.

## Critical Bugs Fixed

### 1. Ghost/Duplicate Session Detection

**Symptom:** User had 8-9 Claude Code sessions but notch showed 16, most labeled "unknown" with idle status and 0k tokens.

**Root Cause:** `scan_processes()` creates sessions for every Claude window AND process it discovers. Hook-detected sessions had `pid=0` (hooks don't send PID), so process scanning never matched them — creating duplicate entries for the same Claude instance.

**Fix (sessions.py):**
- Added `Session.is_displayable` property — process-detected sessions with no project are hidden from UI
- Added `SessionManager._projects_match()` — matches by full path first, then basename (case-insensitive)
- `handle_event()` now checks for existing process-detected sessions matching the hook's project_dir and merges them (steals PID, deletes ghost)
- `scan_processes()` tries to assign PIDs to existing hook sessions before creating new entries
- `cleanup_dead()` more aggressively removes process ghosts with no project (2 min vs 5 min)
- `get_active_sessions()` now filters by `is_displayable`
- `project_name` property falls back to "PID XXXX" instead of "unknown"

### 2. Context Bar Overflow (3019k / 200k)

**Symptom:** Context bars showed absurd numbers like "3019k / 200k" and "5014k / 200k", always solid red.

**Root Cause:** `TokenAggregator.get_session()` returns *lifetime* tokens (total across all context window resets). Comparing this to the single-window context limit (200k) is meaningless — a long session will easily exceed 200k total while the current context window might be half-empty.

**Fix (ui/notch.py):**
- When `display_tokens > context_limit`: show "X.XM total" or "Xk total"
- When within limit: show "Xk / 200k" as before
- Bar stays capped at 100% (was already `min(1.0, ...)`)

**Also fixed in token_aggregator.py:**
- `get_session()` now returns early for `proc-XXXX` session IDs (no JSONL files exist for those)

## Audit Findings NOT Fixed (Accepted/Deferred)

| Finding | Status | Rationale |
|---------|--------|-----------|
| Hook server local injection risk | Accepted | 127.0.0.1 only, low blast radius |
| ExecutionPolicy Bypass in PS1 | Accepted | Standard practice for hook scripts |
| TOCTOU race in lock file | Accepted | OS mutex would be more robust but practical risk is negligible |
| No `__all__` exports | Deferred | Not blocking, can add incrementally |
| No scrollable sessions list | Deferred | max_sessions_shown=6 works for most users |
| No right-click context menu | Deferred | Nice-to-have, not essential |
| `HAS_KEYBOARD`/`HAS_TOAST` guards | Kept | Protects against broken installs |
